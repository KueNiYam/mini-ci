Mini CI

Operations Wiki

Mini CI runs configured shell commands against directories that are already deployed on the CI machine.

Mini CI는 CI 머신에 이미 배포된 디렉터리에서 설정된 shell command를 실행합니다.

What Mini CI Does

Mini CI is the execution and visibility layer. It does not choose source revisions, clone repositories, copy files, or promote releases.

Mini CI는 실행과 가시성 계층입니다. 소스 revision 선택, repository clone, 파일 복사, release promotion은 담당하지 않습니다.
Owns CI execution Project registration, run creation, command execution, logs, status, and rerun actions.
Leaves deployment outside Your deploy process prepares the directory, then calls Mini CI over HTTP.

Operating Flow

The HTTP trigger is the boundary between deployment work and CI execution.

HTTP trigger는 배포 작업과 CI 실행을 나누는 경계입니다.

Deployment side

1

Source is ready

Select the version or worktree that should be tested.

2

Directory is deployed

Copy files to the CI machine under the project root.

3

HTTP trigger

Send project name, worktree path, and run date to Mini CI.

Mini CI side

4

Job is queued

The API returns 202 Accepted with jobId.

5

Commands run

Background execution streams stdout and stderr into the job log.

6

Result is visible

Dashboard shows status, logs, history, and rerun actions.

Deployment can be changed independently as long as it prepares the directory and calls the Run API.

디렉터리를 준비하고 Run API를 호출하기만 하면, 배포 방식은 Mini CI와 독립적으로 바꿀 수 있습니다.

Run Lifecycle

A run is created first, then executed in the background. Logs can be inspected while the job is still running.

run은 먼저 생성되고 background에서 실행됩니다. job이 실행 중이어도 로그를 확인할 수 있습니다.
queued Job was created and is waiting to start.
running Commands are executing and logs are being written.
success All configured commands exited with code 0.
failed A command failed and the failed step was recorded.
interrupted The server restarted while a job was marked running.

Start The Server

Start the server on the CI machine. Network access is the security boundary because Mini CI does not include built-in authentication.

CI 머신에서 서버를 시작합니다. Mini CI는 내장 인증이 없으므로 네트워크 접근 가능 여부가 보안 경계입니다.
MINI_CI_PORT="<port>"
bin/mini-ci start --host 0.0.0.0 --port "${MINI_CI_PORT}"

Project paths are discovered by project name under ~/.codex/worktrees by default. Set MINI_CI_PROJECT_ROOT to use a different base directory.

프로젝트 경로는 기본적으로 ~/.codex/worktrees 아래에서 프로젝트명으로 자동 탐지합니다. 다른 기준 디렉터리를 쓰려면 MINI_CI_PROJECT_ROOT를 설정합니다.

Client Process API

After deployment finishes, the client process should refresh project settings by project name and then start a run for the selected worktree.

배포가 끝나면 클라이언트 프로세스가 프로젝트명으로 설정을 갱신하고 선택한 worktree에 대해 run을 시작해야 합니다.
MINI_CI_URL="http://<mini-ci-host>:<port>"
PROJECT_NAME="<project-name>"
WORKTREE_ID="<worktree-id>"
WORKTREE_PATH="${WORKTREE_ID}/${PROJECT_NAME}"
RUN_DATE="$(date +%Y%m%d%H%M%S)"
COMMANDS='["npm test"]'

curl -X POST "${MINI_CI_URL}/api/admin/projects" \
  -H "Content-Type: application/json" \
  -d "{\"name\":\"${PROJECT_NAME}\",\"commands\":${COMMANDS}}"

curl -X POST "${MINI_CI_URL}/api/projects/${PROJECT_NAME}/runs" \
  -H "Content-Type: application/json" \
  -d "{\"worktreePath\":\"${WORKTREE_PATH}\",\"runDate\":\"${RUN_DATE}\"}"

The Admin API discovers matching directories under the project root. The Run API returns 202 Accepted with jobId. Use GET /api/jobs/:id or the Dashboard to inspect status.

Admin API는 프로젝트 기준 디렉터리 아래에서 일치하는 디렉터리를 자동 탐지합니다. Run API는 202 Accepted와 jobId를 반환합니다. 상태는 GET /api/jobs/:id 또는 Dashboard에서 확인합니다.

Dashboard

The dashboard is the main inspection surface. History is grouped by project and worktree, and each run appears as one date row.

Dashboard는 주요 확인 화면입니다. 실행 이력은 project와 worktree 기준으로 묶이고, 각 run은 하나의 날짜 행으로 표시됩니다.
  • Select a run row to see the selected run summary.
  • Open run detail to inspect the console log without losing the history view.
  • Use rerun only after confirming the target worktree and command list.
run 행을 선택해 요약을 보고, 상세 페이지에서 console log를 확인하고, rerun 전에는 실행 대상과 command를 확인합니다.

Troubleshooting

Start with the project name, worktree path, job status, and log output.

문제가 생기면 project name, worktree path, job status, log output부터 확인합니다.
404 project not found The project was not registered or the project name does not match.
400 bad worktree path The requested worktree path is outside the project root or not registered.
interrupted The server restarted while the job was marked as running.
