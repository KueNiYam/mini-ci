Mini CI Runbook

Run checks after your deploy process prepares a directory.

Mini CI is a lightweight execution layer for local self-hosted CI. It receives HTTP run requests, executes configured shell commands in prepared directories, streams logs, and shows results on the Dashboard.

What it runs Commands you register for a project.
Where it runs Inside an already deployed worktree directory.
What it shows Status, history, console logs, and rerun controls.

Purpose

Use this WIKI when connecting a project to Mini CI.

This document is an operator runbook. It helps a first-time user understand the model, start the server, register a project, trigger a run from a client process, and read the Dashboard.

Mini CI owns Project settings, job creation, background execution, logs, status, and rerun.
Mini CI does not own Git clone, source selection, file copy, release promotion, or deployment policy.
Security boundary There is no built-in login. Expose the server only on a trusted network.

Directory model

Projects are grouped by name under a worktree root.

By default, Mini CI scans ~/.codex/worktrees. A project named my-app can have multiple worktree directories, and each run can target one of them.

~/.codex/worktrees ├─ a1b2/ │ └─ my-app/ ├─ c3d4/ │ └─ my-app/ └─ e5f6/ └─ another-app/
PROJECT_NAME The project folder name, such as my-app.
WORKTREE_ID The parent folder that identifies a prepared worktree, such as a1b2.
WORKTREE_PATH The relative path Mini CI should run, such as a1b2/my-app.
RUN_DATE The date label shown as one row in Dashboard history.

First setup

Start the server, then register the project from Admin.

Start Mini CI on the CI machine. Use MINI_CI_PROJECT_ROOT only when the default worktree root is not the right place for your prepared directories.

MINI_CI_PORT="<port>"
bin/mini-ci start --host 0.0.0.0 --port "${MINI_CI_PORT}"
1
Open Admin.

Go to http://<mini-ci-host>:<port>/admin.

2

Mini CI discovers matching directories under the project root.

3
Save commands.

Use one shell command per line, such as npm test.

Client process integration

The client process should refresh settings and trigger a targeted run.

The Dashboard manual run is useful for inspection, but the normal production flow should call the API after deployment completes.

MINI_CI_URL="http://<mini-ci-host>:<port>"
PROJECT_NAME="<project-name>"
WORKTREE_ID="<worktree-id>"
WORKTREE_PATH="${WORKTREE_ID}/${PROJECT_NAME}"
RUN_DATE="$(date +%Y%m%d%H%M%S)"
COMMANDS='["npm test"]'

# Refresh project directories and commands.
curl -X POST "${MINI_CI_URL}/api/admin/projects" \
  -H "Content-Type: application/json" \
  -d "{\"name\":\"${PROJECT_NAME}\",\"commands\":${COMMANDS}}"

# Start one run for the selected worktree.
curl -X POST "${MINI_CI_URL}/api/projects/${PROJECT_NAME}/runs" \
  -H "Content-Type: application/json" \
  -d "{\"worktreePath\":\"${WORKTREE_PATH}\",\"runDate\":\"${RUN_DATE}\"}"

The Run API returns 202 Accepted with jobId. Command execution continues in the background, and log output is appended while the job runs.

Run status

Status tells you where the job stopped.

Most failures are command failures. interrupted is different: it means the server restarted while a job was marked as running.

queued Job exists but has not started yet.
running Commands are executing and logs are streaming.
success All commands exited with code 0.
failed A command exited non-zero; check failed step and log.
interrupted Server restart recovered a stale running job.

Troubleshooting

Start from project name, worktree path, command, and log.

The fastest debug path is to confirm what Mini CI was asked to run, where it ran, and what the command printed.

404 project not found Register or refresh the project from Admin or POST /api/admin/projects.
400 bad worktree path Check that WORKTREE_PATH is relative to the project root and already registered.
Command failed Open the job detail page and inspect the failed step and console log.
No matching directory Confirm that the deployed folder name matches PROJECT_NAME.
Wrong command saved Open Admin, select the project, edit commands, then save again.
interrupted Rerun after confirming whether the previous command had side effects.

Integration handoff

What to give another automation or AI agent.

When another process prepares the source, give it the project name, target worktree path convention, command list, and Mini CI base URL.

Connect this project to Mini CI.

Project name: <project-name>
Worktree path: <worktree-id>/<project-name>
Commands: ["npm test"]
Mini CI URL: http://<mini-ci-host>:<port>

After deploying files to the worktree path:
1. POST /api/admin/projects with project name and commands.
2. POST /api/projects/:name/runs with worktreePath and runDate.
3. Report the returned jobId and Dashboard URL.

API reference

Use the API reference after the flow is clear.

The core integration uses only two POST requests. The remaining endpoints are for listing projects, reading jobs, logs, and rerunning.

Project and run POST /api/admin/projects
POST /api/projects/:name/runs
Read status GET /api/jobs/:id
GET /api/jobs/:id/logs
Dashboard lists GET /api/projects
GET /api/jobs
Rerun POST /api/jobs/:id/rerun
