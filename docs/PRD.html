Mini CI Product Requirements

Mini CI provides a small HTTP-triggered CI server for directories that are already deployed on the CI machine.

Mini CI는 CI 머신에 이미 배포된 디렉터리를 대상으로 동작하는 작은 HTTP trigger 기반 CI 서버입니다.

Goal

Let a user deploy source files with any tool, then run configured shell commands from the Mini CI dashboard or API.

사용자가 원하는 도구로 소스 파일을 배포한 뒤, Mini CI 대시보드나 API에서 설정된 shell command를 실행할 수 있게 합니다.

Core Workflow

deploy files to CI machine
-> POST /api/admin/projects
-> POST /api/projects/:name/runs with worktreePath and runDate
-> receive 202 Accepted with jobId
-> run commands in the registered directory in background
-> stream logs and store status
-> view result in dashboard

Interfaces

Projects are registered from /admin, and client deployment processes can refresh them with the selected worktree path through POST /api/admin/projects.

프로젝트는 /admin에서 등록하고, 클라이언트 배포 프로세스는 선택한 worktree path를 POST /api/admin/projects로 갱신할 수 있습니다.

Project paths are discovered by project name under the project base directory, which defaults to ~/.codex/worktrees and can be overridden with MINI_CI_PROJECT_ROOT.

프로젝트 경로는 project base 아래에서 project name으로 자동 탐지하며, 기본값은 ~/.codex/worktrees이고 MINI_CI_PROJECT_ROOT로 바꿀 수 있습니다.

The Admin page displays the project base as a home-relative path and groups matching directories under one project name.

Admin 페이지는 project base를 홈 상대경로로 표시하고, 일치하는 디렉터리들을 하나의 project name 아래에 묶어 보여줍니다.

Runs are triggered with POST /api/projects/:name/runs without authentication. The body should include worktreePath and runDate. The API returns 202 Accepted with jobId, then command execution continues in the background.

run은 인증 없이 POST /api/projects/:name/runs 요청으로 시작합니다. body에는 worktreePath와 runDate를 포함합니다. API는 202 Accepted와 jobId를 반환하고, command 실행은 background에서 계속됩니다.

Security

Mini CI does not include built-in authentication. Network access is the security boundary.

Mini CI는 내장 인증을 제공하지 않습니다. 네트워크 접근 가능 여부가 보안 경계입니다.
