Mini CI 운영 Runbook

배포 프로세스가 디렉터리를 준비한 뒤 검증을 실행합니다.

Mini CI는 로컬 self-hosted CI를 위한 가벼운 실행 계층입니다. HTTP run 요청을 받고, 준비된 디렉터리에서 설정된 shell command를 실행하며, 로그를 streaming으로 저장하고 Dashboard에 결과를 보여줍니다.

무엇을 실행하나 프로젝트에 등록한 command를 실행합니다.
어디에서 실행하나 이미 배포된 worktree 디렉터리 안에서 실행합니다.
무엇을 보여주나 상태, 이력, console log, rerun control을 보여줍니다.

목적

프로젝트를 Mini CI에 연결할 때 이 WIKI를 사용합니다.

이 문서는 운영 runbook입니다. 처음 보는 사용자가 모델을 이해하고, 서버를 시작하고, 프로젝트를 등록하고, 클라이언트 프로세스에서 run을 호출하고, Dashboard를 읽을 수 있게 하는 것이 목적입니다.

Mini CI가 담당하는 것 프로젝트 설정, job 생성, background 실행, log, status, rerun입니다.
Mini CI가 담당하지 않는 것 Git clone, source 선택, 파일 복사, release promotion, 배포 정책입니다.
보안 경계 내장 로그인은 없습니다. 신뢰할 수 있는 네트워크에서만 서버를 노출하세요.

디렉터리 모델

프로젝트는 worktree root 아래에서 이름으로 묶입니다.

Mini CI는 기본적으로 ~/.codex/worktrees를 탐색합니다. my-app이라는 프로젝트는 여러 worktree 디렉터리를 가질 수 있고, 각 run은 그중 하나를 대상으로 실행할 수 있습니다.

~/.codex/worktrees ├─ a1b2/ │ └─ my-app/ ├─ c3d4/ │ └─ my-app/ └─ e5f6/ └─ another-app/
PROJECT_NAME 프로젝트 폴더명입니다. 예: my-app.
WORKTREE_ID 준비된 worktree를 식별하는 부모 폴더입니다. 예: a1b2.
WORKTREE_PATH Mini CI가 실행할 상대 경로입니다. 예: a1b2/my-app.
RUN_DATE Dashboard history에서 하나의 날짜 행으로 표시되는 값입니다.

클라이언트 프로세스 연동

클라이언트 프로세스는 설정을 갱신하고 특정 worktree run을 시작해야 합니다.

Dashboard 수동 실행은 확인용으로 유용하지만, 일반적인 운영 흐름에서는 배포가 끝난 뒤 클라이언트 프로세스가 API를 호출해야 합니다.

MINI_CI_URL="http://<mini-ci-host>:<port>"
PROJECT_NAME="<project-name>"
WORKTREE_ID="<worktree-id>"
WORKTREE_PATH="${WORKTREE_ID}/${PROJECT_NAME}"
RUN_DATE="$(date +%Y%m%d%H%M%S)"
COMMANDS='["npm test"]'

# Project directory와 command를 갱신합니다.
curl -X POST "${MINI_CI_URL}/api/admin/projects" \
  -H "Content-Type: application/json" \
  -d "{\"name\":\"${PROJECT_NAME}\",\"commands\":${COMMANDS}}"

# 선택한 worktree에 대해 run을 시작합니다.
curl -X POST "${MINI_CI_URL}/api/projects/${PROJECT_NAME}/runs" \
  -H "Content-Type: application/json" \
  -d "{\"worktreePath\":\"${WORKTREE_PATH}\",\"runDate\":\"${RUN_DATE}\"}"

Run API는 202 Accepted와 jobId를 반환합니다. Command 실행은 background에서 계속되고, job 실행 중 log output이 계속 누적됩니다.

Run status

Status는 job이 어디에서 멈췄는지 알려줍니다.

대부분의 실패는 command 실패입니다. interrupted는 다릅니다. 서버가 재시작될 때 job이 running으로 남아 있었음을 뜻합니다.

queued Job은 생성됐지만 아직 시작하지 않았습니다.
running Command가 실행 중이고 log가 streaming되고 있습니다.
success 모든 command가 exit code 0으로 종료됐습니다.
failed Command가 non-zero로 종료됐습니다. failed step과 log를 확인합니다.
interrupted 서버 재시작으로 stale running job이 복구된 상태입니다.

Troubleshooting

Project name, worktree path, command, log부터 확인합니다.

가장 빠른 디버깅 경로는 Mini CI가 무엇을 실행하라는 요청을 받았는지, 어디에서 실행했는지, command가 무엇을 출력했는지 확인하는 것입니다.

404 project not found Admin 또는 POST /api/admin/projects로 project를 등록하거나 갱신합니다.
400 bad worktree path WORKTREE_PATH가 project root 기준 상대 경로이고 등록된 경로인지 확인합니다.
Command failed Job detail page에서 failed step과 console log를 확인합니다.
일치하는 디렉터리 없음 배포된 폴더명이 PROJECT_NAME과 일치하는지 확인합니다.
잘못된 command 저장 Admin에서 project를 선택하고 command를 수정한 뒤 다시 저장합니다.
interrupted 이전 command의 side effect 가능성을 확인한 뒤 rerun합니다.

연동 전달 정보

다른 자동화나 AI agent에게 무엇을 주면 되나.

다른 프로세스가 소스를 준비한다면, project name, target worktree path 규칙, command list, Mini CI base URL을 알려주면 됩니다.

이 프로젝트를 Mini CI에 연결하세요.

Project name: <project-name>
Worktree path: <worktree-id>/<project-name>
Commands: ["npm test"]
Mini CI URL: http://<mini-ci-host>:<port>

파일을 worktree path에 배포한 뒤:
1. project name과 commands로 POST /api/admin/projects를 호출합니다.
2. worktreePath와 runDate로 POST /api/projects/:name/runs를 호출합니다.
3. 반환된 jobId와 Dashboard URL을 보고합니다.

API reference

흐름을 이해한 뒤 API reference를 봅니다.

핵심 연동에는 POST 요청 두 개만 필요합니다. 나머지 endpoint는 project 조회, job 조회, log 확인, rerun에 사용합니다.

Project and run POST /api/admin/projects
POST /api/projects/:name/runs
Read status GET /api/jobs/:id
GET /api/jobs/:id/logs
Dashboard lists GET /api/projects
GET /api/jobs
Rerun POST /api/jobs/:id/rerun
